# FAST Response Eval Method

This method evaluates whether a response to a Phase 4B FAST benchmark prompt follows FAST/lightwork principles. It is designed for local Mac and closed-network environments.

## Runner Mode

- Default mode is `mock/dry-run`.
- Default mode never calls an external model.
- Optional mode is `KIWI_FAST_EVAL_RUNNER=codex`.
- In optional mode the runner tries the local `codex` CLI through a real subprocess path.
- `KIWI_FAST_EVAL_CODEX_MODEL=<model>` optionally pins the codex model. When set, the runner passes `-m <model>` to `codex exec`; when unset, codex uses its configured default model.
- Codex mode uses feature-detected safety args from `codex exec --help`.
- Codex mode must retain `--ephemeral`, `--sandbox read-only`, and `-m <model>` when the local CLI advertises those options.
- `--ask-for-approval never` is optional safety hardening. The runner includes it only when `codex exec --help` advertises `--ask-for-approval`; otherwise it omits the unsupported flag and records `unsupported_safety_args: ["--ask-for-approval", "never"]`.
- The assertion suite creates a temp PATH fake codex executable so the codex success and failure matrix is verified without depending on a real installed CLI.
- The fake codex executable simulates both supported and unsupported `--ask-for-approval` help output. The supported case verifies argv contains `--ask-for-approval never`; the unsupported case verifies argv omits it and records `unsupported_safety_args`.
- Both fake codex cases verify argv contains the model option when `KIWI_FAST_EVAL_CODEX_MODEL` is set, plus `--ephemeral` and `--sandbox read-only`.
- The codex prompt asks only for a short response sample. It tells codex not to read, edit, patch, or run commands against repository files.
- Unsupported safety args are metadata only, not task fallback events. `fallback_events` records only actual task execution failures such as missing CLI, timeout, nonzero exit, empty output, or subprocess errors.
- If the CLI is missing, times out, or returns an error during task execution, the run records task-level fallback to mock output with a clear fallback reason.
- Runner metadata includes `source_counts`, `fallback_events`, `codex_available`, `codex_model`, safety args, `unsupported_safety_args`, and command args without the response prompt; reviewers should not infer behavior from a single global runner value.
- If `response-eval-results.codex.json` exists, requested runner is `codex`, and `codex_available=true`, the assertion suite requires at least one `response_source=codex` task so an all-`mock_fallback` artifact cannot pass.
- If a codex-requested artifact contains any `response_source=codex` task with `pass=false`, that codex-originated FAIL must be closed in `evaluation-report.md` or `response-eval-codex-fail-closures.md`.
- A codex-originated FAIL closure must mention the `task_id`, every recorded `failure_reasons` entry, and one classification: `accepted prompt improvement`, `accepted rubric improvement`, `accepted benchmark adjustment`, or `not accepted with reason`.
- Codex-requested runs write the normal `response-eval-results.json`/`.md` path and also preserve an audit copy at `response-eval-results.codex.json`/`.md`.
- `response-eval-results.json` uses stable generated_at behavior: equivalent reruns reuse the previous timestamp so timestamp-only dirty diffs are not created. Set `KIWI_FAST_EVAL_REFRESH_GENERATED_AT=1` only when intentionally refreshing the timestamp.

## GPT-5.5 xhigh evaluator packet

A GPT-5.5 xhigh evaluator should inspect:

- the Phase 4B `final_prompt` used for each task;
- the response sample generated by the runner;
- rubric scores and notes;
- failure reasons linked to rubric keys;
- failure patterns and accepted improvements.

No external GPT-5.5 or Qwen3.5 call is required for this local loop.

## Scoring Rubric

Each criterion is scored `0-2`.

- `project_info_current_file_verification`: Project Info 먼저 읽기/현재 파일 검증 언급.
- `todowrite_planning`: TodoWrite/todo_write is named as the planning mechanism before substantive work.
- `short_korean_plan`: short Korean plan, preferably a concise numbered plan.
- `minimal_scope`: minimal scope and minimal diff discipline.
- `focused_verification`: focused verification command or fallback check.
- `stop_question_conditions`: stop/question condition recognition.
- `forbidden_leakage_absent`: forbidden leakage detection for FAST-prohibited delegation and sizing language.

Total score is 14. A response passes when total score is at least 12 and `forbidden_leakage_absent` is 2.

## Mock Samples

Mock mode intentionally includes at least two intentionally weak response samples. These prove the evaluator can produce FAIL, not only artificial PASS.

Expected weak failure patterns:

- Missing Project Info and current files evidence.
- No short Korean plan.
- Missing TodoWrite/todo_write planning.
- Broad or vague implementation scope.
- Missing focused verification.
- Missing stop/question conditions.
- Forbidden leakage in FAST response language.

## Accepted Improvements

- PASS samples use Project Info first, current files verification, TodoWrite planning, short Korean plan, minimal scope, focused verification, and stop/question language.
- FAIL samples are retained as regression fixtures.
- Forbidden leakage is detected from the response sample text and linked to `forbidden_leakage_absent`.
